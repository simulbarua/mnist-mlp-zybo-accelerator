# Vivado Implementation Guide — MLP Accelerator on Zybo Z7-10

## Overview

**Network architecture:** 784 → FC1(64, ReLU) → FC2(32, ReLU) → FC3(10, argmax)  
**Quantization:** INT8 symmetric post-training  
**Target board:** Digilent Zybo Z7-10 (XC7Z010-1CLG400C)  
**PL clock:** 100 MHz (FCLK_CLK0 from Zynq PS)  
**Inference time:** ~1.58 ms per image

## System Architecture

```
Zynq PS (ARM Cortex-A9)
   │
   │  M_AXI_GP0 (32-bit AXI, 100 MHz)
   │
   ▼
AXI SmartConnect (1 master → 3 slaves)
   │
   ├──► AXI BRAM Controller 0  ──► BMG 0 Port A (32-bit, write path from PS)
   │                                BMG 0 Port B (8-bit, enb-controlled) ──► mlp_engine param_bram_*
   │                                [Param BRAM — 64 KB: INT8 weights + INT32 biases]
   │
   ├──► AXI BRAM Controller 1  ──► BMG 1 Port A (32-bit, write path from PS)
   │                                BMG 1 Port B (8-bit, enb-controlled) ──► mlp_engine input_bram_*
   │                                [Input BRAM — 1 KB: 784 INT8 pixel bytes]
   │
   └──► mlp_top AXI-Lite slave
            │
            ├── axilite_ctrl  (CTRL / STATUS / RESULT registers)
            └── mlp_engine    (serial MAC FSM — INT32 accumulator)
```

### Data Flow

1. **Boot:** ARM writes weights + biases to Param BRAM via AXI BRAM Ctrl 0 (`mlp_bram_init()`)
2. **Per inference:**  
   a. ARM writes 784 pixel bytes to Input BRAM via AXI BRAM Ctrl 1  
   b. ARM writes `0x1` to CTRL register → triggers `start` pulse to engine  
   c. Engine reads weights byte-by-byte and biases 4 bytes (INT32, little-endian) via Port B; reads input pixels byte-by-byte via Port B  
   d. Engine computes FC1 → ReLU → FC2 → ReLU → FC3 → argmax  
   e. Engine asserts `done` pulse, writes 4-bit class to RESULT register  
   f. ARM polls STATUS[0] (done), then reads RESULT[3:0]

### Address Map

| Peripheral | Base Address | Size |
|---|---|---|
| Param BRAM (AXI BRAM Ctrl 0) | `0x4000_0000` | 64 KB |
| Input BRAM (AXI BRAM Ctrl 1) | `0x4001_0000` | 4 KB |
| mlp_top AXI-Lite | `0x4002_0000` | 4 KB |

### AXI-Lite Register Map

| Offset | Name | Dir | Bits | Description |
|---|---|---|---|---|
| `+0x00` | CTRL | W | `[0]` | Write `1` to start inference (auto-clears) |
| `+0x04` | STATUS | R | `[0]` | Done flag (latched, clears on read) |
| `+0x04` | STATUS | R | `[1]` | Busy flag (high while engine is running) |
| `+0x08` | RESULT | R | `[3:0]` | Predicted class 0–9 |

## Part 2 — Package mlp_top as a Custom IP

The block design requires `mlp_top` to be packaged so Vivado can infer its AXI-Lite interface.

1. **Tools → Create and Package New IP**
2. Choose **Package a specified directory** → point to the `hardware/rtl/` folder
3. Set name: `mlp_accelerator`, version: `1.0`
4. In the **IP Packager**:
   - **Ports and Interfaces tab:** Vivado should auto-infer the `S_AXI` (AXI4-Lite) interface from the `s_axi_*` port names. If not, right-click → **Add Interface** → AXI4-Lite → assign ports manually.
   - The `param_bram_*` and `input_bram_*` ports are left as plain ports; they connect directly to BMG Port B nets in the block design.
5. **Review and Package** → **Package IP**

| Setting | Value |
|---|---|
| Write Width | **8** |
| Read Width | **8** |

| Primitives Output Register | **Unchecked** (DO_REG=0 → 1-cycle read latency) |
| Enable Port Type | **Use ENB Pin** |

> **Why Use ENB Pin:** `mlp_engine` drives `param_bram_en` and only asserts it
> when it presents a valid read address. Selecting "Use ENB Pin" connects this
> signal directly to the BRAM's `enb` port, so the BRAM only latches a new
> address when the engine requests it. "Always Enabled" would expose no `enb`
> port and cause the BRAM to continuously track the address bus — functionally
> it still works but the `param_bram_en` output from the engine would be
> unconnected and the design intent is lost.

### 6.4 main.c Template

> **Critical:** The Input BRAM must receive **INT8 normalized pixels**, not raw
> `uint8` values.  The model was trained on inputs normalized by
> `(pixel/255 − 0.1307) / 0.3081`.  Feeding raw pixels to weights calibrated
> for that range causes the FC1 accumulator to overflow by ~500×, collapsing
> accuracy to ~10% (random).  The normalization step below fixes this.

```c
#include <stdint.h>
#include <math.h>
#include "xil_io.h"
#include "xil_printf.h"
#include "mlp_bram_init.h"
#include "weights_biases.h"   /* provides MLP_INPUT_MEAN, MLP_INPUT_STD, MLP_INPUT_SCALE */

/* AXI-Lite register addresses — must match Vivado Address Editor */
#define MLP_AXILITE_BASE  0x40020000UL
#define MLP_CTRL_ADDR    (MLP_AXILITE_BASE + 0x00UL)
#define MLP_STATUS_ADDR  (MLP_AXILITE_BASE + 0x04UL)
#define MLP_RESULT_ADDR  (MLP_AXILITE_BASE + 0x08UL)

/* Input BRAM base address (AXI BRAM Controller 1) */
#define INPUT_BRAM_BASE  0x40010000UL

/* Clamp x to [lo, hi] */
static inline int clamp_int(int x, int lo, int hi) {
    return x < lo ? lo : (x > hi ? hi : x);
}

/*
 * normalize_pixels()
 * Convert 784 raw uint8 pixels to INT8 quantized values.
 *   1. Normalize:  x_f = (pixel/255.0 - MEAN) / STD
 *   2. Quantize:   x_i = round(x_f / INPUT_SCALE), clipped to [-128, 127]
 *   3. Write as two's-complement bytes to the Input BRAM.
 *
 * The RTL sign-extends each byte when reading Port B, so negative values
 * are handled correctly in hardware.
 */
static void normalize_pixels(const uint8_t *pixels, uint32_t n)
{
    volatile uint8_t *bram = (volatile uint8_t *)INPUT_BRAM_BASE;
    for (uint32_t i = 0; i < n; i++) {
        float x_f = ((float)pixels[i] / 255.0f - MLP_INPUT_MEAN) / MLP_INPUT_STD;
        int   x_i = (int)roundf(x_f / MLP_INPUT_SCALE);
        bram[i]   = (uint8_t)(int8_t)clamp_int(x_i, -128, 127);
    }
}

int main(void) {

    /* ── Step 1: Load weights/biases into Param BRAM (once at boot) ── */
    xil_printf("Loading MLP parameters into BRAM...\r\n");
    mlp_bram_init();
    if (mlp_bram_readback_check() != 0) {
        xil_printf("ERROR: BRAM readback failed. Check MLP_BRAM_BASE_ADDR.\r\n");
        return -1;
    }
    xil_printf("BRAM loaded OK.\r\n");

    /* ── Step 2: Normalize and write 784 pixels to Input BRAM ────── */
    extern uint8_t my_image[784];   /* raw uint8 pixels, 0–255 */
    normalize_pixels(my_image, 784);

    /* ── Step 3: Trigger inference ────────────────────────────────── */
    Xil_Out32(MLP_CTRL_ADDR, 0x00000001UL);

    /* ── Step 4: Poll STATUS[0] (done flag, read-to-clear) ───────── */
    while ((Xil_In32(MLP_STATUS_ADDR) & 0x1U) == 0U)
        ;

    /* ── Step 5: Read predicted class ────────────────────────────── */
    uint32_t cls = Xil_In32(MLP_RESULT_ADDR) & 0xFU;
    xil_printf("Predicted class: %d\r\n", (int)cls);

    return 0;
}
```

## Part 7 — Resource and Timing Summary

| Metric | Value |
|---|---|
| Inference time @ 100 MHz | ~1.59 ms |
| FC1 cycles: 64 × (12 bias + 784×3 MAC) | ~151 296 |
| FC2 cycles: 32 × (12 bias +  64×3 MAC) | ~6 528 |
| FC3 cycles: 10 × (12 bias +  32×3 MAC) | ~1 080 |
| Bias read overhead | 4 bytes × 3 cycles = 12 cycles/neuron (INT32, little-endian) |
| Param BRAM capacity needed | ~52.0 KB |
| BRAM blocks used (Param) | ~13 × RAMB36 |
| BRAM blocks used (Input) | 1 × RAMB36 |
| DSPs used | 1 (INT8 × INT8 multiplier) |
| Weight data type | INT8 (4 packed per 32-bit BRAM word) |
| Bias data type | INT32, signed (1 per 32-bit BRAM word) |
| MAC accumulator | INT32 |
| Inter-layer requantization | `act_int8 = clip(ReLU(acc >> k), 0, 127)` — shifts in `mlp_params.vh` |
| Inter-layer activations | ReLU + saturating clip to [0, 127] |

## Troubleshooting Checklist

| Symptom | Likely cause | Fix |
|---|---|---|
| `mlp_bram_readback_check()` returns mismatches | Wrong `MLP_BRAM_BASE_ADDR` | Match to Vivado Address Editor |
| STATUS never shows done | Engine start pulse not reaching PL, or bitstream mismatch | Re-export hardware, rebuild Vitis project |
| Wrong inference result | Input pixels not written to correct BRAM address | Verify `INPUT_BRAM_BASE` = address of AXI BRAM Ctrl 1 |
| Synthesis fails on `mlp_engine.v` | Vivado version < 2019.2 | Update or remove `integer` loop variable in reset block |
| BMG Port B data always 0 | DO_REG mismatch or EN not asserted | Confirm BMG config: Primitives Output Register = unchecked, Enable Port Type = Always Enabled |
| Timing not met | Clock too fast | Reduce FCLK_CLK0 to 50 MHz (adjust XDC constraint) |

## Regenerating Weights After Retraining

```bash
cd software/
python train_and_export.py
# Outputs:
#   hardware/firmware/weights_biases.h  — rebuild Vitis project to pick up new values
#   hardware/rtl/mlp_params.vh          — rebuild Vivado project to pick up new shifts
#   coe_files/                          — for Vivado BRAM initialisation (if needed)
```

Two things change after retraining:

1. **Vitis (C firmware):** Vitis recompiles `mlp_bram_init.c` automatically on the next build because it includes `weights_biases.h`. New weights, INT32 biases, input scale, and shift constants are picked up without any manual edits.

2. **Vivado (RTL):** `rtl/mlp_params.vh` is updated with the new `MLP_REQ_SHIFT_FC1` and `MLP_REQ_SHIFT_FC2` values. Re-synthesise and re-implement the Vivado project (Synthesis → Implementation → Generate Bitstream) to burn the new shifts into the PL.

> **Why the shifts matter:** After each FC layer MAC, the INT32 accumulator is ≈10 000–50 000× larger than the float equivalent because both weights and inputs are quantised to small INT8 values. Without a right-shift before the ReLU clip, every positive neuron outputs 127 (binary activation), collapsing accuracy from ~97% to ~27%. The right-shift `k = round(log2(act_scale / (w_scale × x_scale)))` is computed from calibrated activation statistics and varies with each training run.
